# MaxKB 接口完成度审计表

> 目标：从第三方稳定对接角度，评估当前 MaxKB 开放能力是否足够，以及哪些能力必须补齐。

## 结论摘要

当前 MaxKB 已经具备**第三方对接的生产级基础能力**。结合当前实现与端到端验证结果，第三方封板所需的核心链路已经打通；剩余能力主要是增强项，而非当前封板阻断项。

当前建议：

1. 保持 `MaxKB` 核心逻辑稳定，不再继续扩展核心改造
2. 第三方接入统一放在 `open_api` 与 Go 层完成
3. `auth/token` 不作为当前第三方主流程必需能力，由 Go 统一持有 API Key 更稳妥
4. 文件上传、问答、会话链路已可稳定使用，后续优化主要集中在返回语义与治理能力上
5. 第三方用户/租户映射建议放在 Go，不建议大改 MaxKB

## 三、第三方稳定对接角度，必须补的能力

### 必须补齐 1：问答真实执行链
当前 `chat/completions` 不能停留在 stub。

必须完成：
- 输入校验
- 真实推理调用
- 检索知识库
- 生成答案
- 保存会话和消息
- 返回标准化结构

### 必须补齐 2：问答返回协议标准化
建议统一成：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "session_id": "...",
    "message_id": "...",
    "answer": "...",
    "sources": [],
    "usage": {},
    "finish_reason": "stop"
  }
}
```

### 必须补齐 3：文件上传格式兜底
建议明确：
- 支持格式
- 不支持格式
- 失败原因
- 上传状态

### 必须补齐 4：Go 侧的业务映射与权限中台
建议把以下逻辑放 Go：
- 用户登录
- 租户隔离
- 知识库与业务对象映射
- 上传前校验
- 任务轮询
- 权限控制
- 审计日志

## 五、建议的最终对接模式

```text
小程序 / 管理后台 -> Go 后台 -> MaxKB
```

不要：

```text
小程序 -> MaxKB
```

原因：
- MaxKB 的 key 不应暴露给前端
- Go 可以统一管理业务权限
- MaxKB 不需要为每个前端做定制鉴权

## 六、最终审计结论

### 已经足够用的部分
- 健康检查
- 知识库管理
- 文档文本上传
- 文档管理
- 会话管理
- API Key 管理

### 需要补齐的部分
- 问答真实执行链
- 标准化问答返回协议
- 文件上传兜底和格式支持说明
- 第三方用户/租户映射（建议在 Go）

### 建议的落地顺序
1. 先补 `chat/completions` 真实链路
2. 再统一问答返回协议
3. 再完善文件上传支持和状态
4. 最后把 Go 侧业务网关和权限中台接起来

---

## 七、总结

当前 MaxKB 对第三方接入来说：
- **基础能力已具备**
- **生产级稳定对接还差最后一层问答链路和协议标准化**

如果遵循“MaxKB 少改、Go 多做”的原则，最重要的是：
- 让 Go 做中台
- 让 MaxKB 做 AI 引擎
- 让前端只对接 Go
