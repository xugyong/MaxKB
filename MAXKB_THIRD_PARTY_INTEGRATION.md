# MaxKB 第三方对接封板版

本文档定义 MaxKB 与第三方系统（Go 后台 + 小程序前端）的正式对接边界、稳定接口、验收标准与上线门槛。本文用于第三方接入封板评审，重点描述**谁负责什么、哪些接口可依赖、哪些行为必须稳定、哪些属于后续增强项**。

> 约定说明
>
> - 这里的“第三方系统”默认指 **Go 后台 + 小程序前端**。
> - 这里的“MaxKB”默认指 **被第三方调用的 AI/RAG 能力服务**。
> - 本文中的“封板”表示第三方接入链路已满足上线评审条件，但不表示所有业务语义都已达到完全无风险状态。

---

## 1. 对接目标

第三方系统由两部分组成：

- **Go 管理后台**：统一鉴权、权限控制、审计日志、业务编排、调用 MaxKB
- **小程序前端**：仅负责展示与交互，不直接访问 MaxKB

MaxKB 作为内部 AI 能力服务，对外仅由 Go 后台调用。第三方接入的第一原则是：**小程序不持有 MaxKB 凭证，不直接依赖 MaxKB 路由，不直接感知 MaxKB 内部实现。**

## 2. 稳定对接原则

### 2.1 统一网关原则
所有第三方请求必须先到 Go 后台，再由 Go 后台调用 MaxKB。

### 2.2 凭证隔离原则
MaxKB API Key 只能保存在服务端，不得下发到小程序或浏览器。

### 2.3 身份分层原则
需要区分两类身份：

- **系统身份**：Go 后台调用 MaxKB 时使用的服务凭证
- **终端用户身份**：小程序用户身份，由 Go 后台映射到 MaxKB 会话

### 2.4 最小改动原则
MaxKB 保留为 AI/RAG 能力服务，业务权限、租户隔离、用户映射、审计日志尽量放在 Go 后台。

### 2.5 稳定契约优先原则
第三方接入依赖的是“接口契约”和“返回语义”，不是内部模型、缓存或流水线细节。只要对外契约不变，内部实现可以优化，但不得随意改变字段语义、错误码含义和会话关联规则。

## 3. 稳定接口边界

### 3.1 MaxKB 对 Go 暴露的稳定接口
以下接口应视为**对接目标**，Go 后台可以依赖其业务能力，但实际是否完全可用、路径是否一致、字段是否一致，需要以当前 MaxKB 版本联调确认：

- 健康检查
- 知识库创建 / 查询 / 更新 / 删除
- 文档上传 / 列表 / 详情 / 删除 / 重处理
- 应用 API Key 管理
- 问答接口
- 会话列表 / 消息列表 / 会话删除

### 3.2 Go 对小程序暴露的稳定业务接口
以下由 Go 自行定义并稳定对外：

- 创建知识库
- 上传文档
- 查询文档状态
- 发起问答
- 查询会话历史
- 查询业务可见的知识库列表

### 3.3 不作为外部契约的内容
以下内容不作为稳定契约，不建议第三方直接依赖，也不应写入前端业务逻辑：

- MaxKB 内部模型表结构
- 任务队列实现细节
- 文档解析器内部状态
- 会话复开内部逻辑
- 缓存命中与否
- 调试模式与生产模式的内部差异

## 4. 认证与授权

### 4.1 Go 调用 MaxKB
Go 后台调用 MaxKB 时，使用：

```text
Authorization: Bearer <MAXKB_API_KEY>
```

### 4.2 小程序调用 Go
小程序只使用 Go 后台自己的登录态：

- JWT
- Session Cookie
- 小程序 code 换 session
- 其他你们已有的登录方式

### 4.3 权限分层
建议至少分三层：

- 管理员：创建知识库、上传文档、管理 API Key
- 普通用户：问答、查看自己的会话
- 业务租户：只能看到自己授权范围内的数据

### 4.4 认证链路边界
- 小程序只认 Go 自己的登录态
- Go 认自己的用户体系和租户体系
- Go 调 MaxKB 时只使用服务端持有的 MaxKB 凭证
- 不允许前端直接透传 MaxKB API Key
- 不允许前端根据 MaxKB 返回字段决定权限

---

## 5. 数据映射要求

Go 后台必须保存以下映射：

### 5.1 知识库映射
- 业务知识库 ID
- MaxKB `workspace_id`
- MaxKB `knowledge_id`
- 名称
- 状态
- 创建者

### 5.2 应用映射
- 业务应用 ID
- MaxKB `application_id`
- MaxKB API Key ID
- 启用状态

### 5.3 会话映射
- 业务用户 ID
- MaxKB `chat_id`
- MaxKB `message_id`
- 最近一次问答内容
- 最近一次回答内容
- 会话所属应用
- 会话所属知识库（如有）
- 会话来源（小程序 / 管理后台 / 其他）
- 会话状态

### 5.4 建议补充字段
为了后续审计和排障，Go 后台建议再保存：
- 请求 ID / trace ID
- 创建来源 IP
- 第一次回答时间
- 最后一次更新时间
- 失败原因（如有）

## 6. 初始化前提

封板前必须确保以下条件满足：

1. MaxKB 服务可正常启动
2. 至少存在一个可用的 `workspace`
3. 至少存在一个已发布且已绑定模型的 `application`
4. 至少存在一个可用的 `application api key`
5. 至少存在一个可用的 embedding 模型
6. 知识库创建接口可用
7. 文档上传接口可用
8. 问答接口可用
9. 会话列表 / 详情 / 消息查询可用
10. 生产环境的超时、告警、日志链路已确认

## 7. 推荐调用流程

### 7.1 知识库创建
```text
管理员 -> Go 后台 -> MaxKB 创建知识库 -> Go 保存 knowledge_id
```

### 7.2 文档上传
```text
管理员 -> Go 后台 -> MaxKB 上传文档 -> Go 轮询状态 -> 前端展示结果
```

### 7.3 用户问答
```text
小程序 -> Go 后台 -> MaxKB chat/completions -> Go 返回答案与来源
```

### 7.4 会话历史
```text
小程序 -> Go 后台 -> 查询 Go 保存的映射 -> 必要时补查 MaxKB
```

说明：
- 正常情况下会话历史应优先以 Go 保存的数据为准
- 必要时可以补查 MaxKB，用于排障或补偿
- 不应让小程序直接拼 MaxKB 会话地址或依赖 MaxKB 内部 session id 规则

## 8. 稳定返回约定

### 8.1 知识库创建
成功时必须返回：

- `code = 0`
- `data.workspace_id`
- `data.knowledge_id`
- `data.name`

### 8.2 文档上传
成功时必须返回：

- `code = 0`
- `data.workspace_id`
- `data.knowledge_id`
- `data.document_id`
- `data.status`

### 8.3 问答接口
成功时必须返回：

- `code = 0`
- `data.session_id`
- `data.message_id`
- `data.answer`
- `data.sources`
- `data.usage`

补充约定：
- `sources` 应优先返回知识库来源；若上游没有可用来源，可由服务端兜底返回最小可解释来源，但不应依赖前端自行推断
- `usage` 优先使用上游统计；若上游缺失，可由服务端按统一规则兜底生成
- `finish_reason` 应保留，用于前端或 Go 判断回答结束状态
- `status` 应保留，用于标识成功、待定或异常状态

### 8.4 失败响应
失败时必须保留：

- `code`
- `message`
- `data`

并保证 Go 侧可将其映射为统一错误码。失败响应应尽量保证：
- 错误码语义稳定
- 提示信息可读
- 不泄露内部实现细节
- 可用于日志与告警定位

---

## 9. 统一错误码建议

Go 后台建议统一映射如下：

- `401`：未登录或凭证无效
- `403`：无权限
- `404`：资源不存在
- `500`：MaxKB 内部处理失败
- `502`：MaxKB 上游不可用
- `504`：MaxKB 超时

## 10. 封板验收表

补充说明：
- “已验证”表示该能力在当前方案中被明确纳入，并应按此标准实现
- “待确认”表示需要结合真实 MaxKB 版本、环境配置、数据规模或业务规则复核
- 若某项依赖具体模型、数据或生产配置，则上线前必须在真实环境复核一次

### 10.2 待确认 / 待补强

- [ ] chat 返回的 `answer` 语义是否完全贴近真实业务问答结果
- [ ] `sources` 在复杂场景下的整理、标题展示与脱敏处理是否满足业务要求
- [ ] `usage` 在不同模型与不同回答模式下的统计口径是否需要扩展

## 11. Go 团队实施指南

这一节是给 Go 团队的最终交付说明。目标是让 Go 侧按统一契约完成接入、编排、治理与验收，并保证 MaxKB 核心保持稳定不被业务接入逻辑污染。

### 11.1 角色定位
Go 后台是第三方接入的唯一业务入口，负责：

- 统一鉴权与用户态管理
- 业务权限控制与租户隔离
- MaxKB 能力编排与稳定调用
- 统一错误映射与审计记录
- 对前端提供稳定、不暴露内部实现的接口

MaxKB 仅作为能力服务使用，不承载 Go 侧的业务会话治理、权限治理与审计主逻辑。

### 11.2 系统分层建议
Go 侧建议分为四层：

#### 11.2.1 Handler / Controller
职责：

- 接收请求
- 校验参数
- 校验登录态
- 转交 Service
- 统一返回 JSON

这一层不写 MaxKB 调用细节。

#### 11.2.2 Service
职责：

- 业务编排
- 调用 MaxKB Client
- 控制重试、超时、降级
- 错误码翻译
- 维护业务状态流转

#### 11.2.3 Client
职责：

- 封装 MaxKB HTTP 调用
- 处理 Base URL、Header、超时、重试
- 解析响应与错误
- 统一序列化与反序列化

#### 11.2.4 Repository / DAO
职责：

- 保存业务映射关系
- 保存会话与文档状态
- 保存审计日志与操作记录
- 保存告警与错误归因信息

### 11.3 必须实现的业务能力
Go 团队至少需要实现以下能力：

1. MaxKB Client 封装
2. 鉴权与凭证管理
3. 知识库创建与查询
4. 文档上传与状态查询
5. 问答与会话管理
6. 会话历史与消息历史
7. 审计日志与监控告警
8. 后台管理页面与权限控制

### 11.4 设计原则
#### 11.4.1 凭证只在服务端保存

- MaxKB API Key 只能保存在服务端
- 不允许下发给前端
- 支持轮换、停用、审计

#### 11.4.2 业务身份与 MaxKB 身份分离

- 业务用户 ID 不直接等同于 MaxKB ID
- 会话、文档、知识库需要建立映射表
- 前端只认 Go 侧业务 ID

#### 11.4.3 幂等优先

- 创建知识库、上传文档、发起问答建议支持幂等键
- 失败重试只针对网络与上游不可用场景
- 业务错误不盲目重试

#### 11.4.4 错误统一翻译

- 不把 MaxKB 原始错误直接透传给前端
- 统一转成业务错误码与可读文案
- 明确区分认证失败、权限不足、资源不存在、上游超时、上游异常

#### 11.4.5 可观测性优先

- 每次调用带 request_id / trace_id
- 所有异常落库或落日志
- 关键链路要有告警
- 关键状态必须可追踪

### 11.5 推荐数据模型
至少建议维护以下映射表：

#### 11.5.1 应用映射表
字段建议：

- 业务应用 ID
- MaxKB `application_id`
- MaxKB `workspace_id`
- MaxKB API Key ID
- 是否启用
- 发布状态
- 创建时间 / 更新时间

#### 11.5.2 知识库映射表

## 12. Go 后台推荐 API 约定

这一节是给 Go 团队和小程序团队的统一约定。目标是让小程序只调用 Go，不直接接触 MaxKB；Go 再去封装 MaxKB 的能力和错误语义。

### 12.1 设计原则
- 小程序只调用 Go 的业务接口，不调用 MaxKB Open API
- Go 统一处理登录态、租户、权限、审计、限流、参数校验
- Go 统一持有 MaxKB API Key，不下发给前端
- Go 返回面向业务的稳定 JSON 结构，不透传 MaxKB 内部字段
- Go 对 MaxKB 的错误码做翻译，前端只感知业务错误

```text
┌──────────────────┐
│    小程序页面    │
└────────┬─────────┘
         │ 1. 登录态 / Token
         ▼
┌──────────────────┐
│    Go API 网关   │
└────────┬─────────┘
         │ 2. 参数校验 / 基础鉴权
         ▼
┌──────────────────┐
│   权限校验层     │
└────────┬─────────┘
         │ 3. 用户 / 租户 / 应用映射
         ▼
┌──────────────────┐
│   业务服务层     │
└──────┬───────┬───┘
       │       │
       │       └──────────────▶ ┌──────────────────┐
       │                        │ 会话 / 知识库映射 │
       │                        └──────────────────┘
       │
       ▼
┌──────────────────┐
│  MaxKB Client    │
└────────┬─────────┘
         │ 4. 调用 MaxKB Open API
         ▼
┌──────────────────┐
│  MaxKB Open API  │
└────────┬─────────┘
         │ 5. 访问知识库 / 文档 / 会话
         ▼
┌──────────────────┐
│  MaxKB 能力层    │
└──────────────────┘
```

### 12.2 推荐的统一返回格式
所有 Go 对外接口建议统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "trace_id": "..."
}
```

约定：
- `code = 0` 表示成功
- `message` 为人类可读提示
- `data` 为业务数据
- `trace_id` 用于排查链路问题，建议所有接口都返回

### 12.3 推荐接口清单

#### 12.3.1 问答接口
```text

```
用途：
- 小程序发起一次问答
- Go 内部转调 MaxKB `chat/completions`

请求示例：

```json
{
  "application_id": "019d95c2-7d9a-7183-b237-9ba530a33ebd",
  "question": "请根据知识库回答 XXX",
  "chat_id": "",
  "stream": false,
  "context": {
    "user_id": "u_10001",
    "tenant_id": "t_10001"
  }
}
```

返回示例：

```json
{
  "code": 0,
  "message": "ok",
  "trace_id": "req_...",
  "data": {
    "session_id": "019d996b-f8af-79e0-b3e7-c1372fa4a597",
    "message_id": "019d996c-3e2a-7a51-8a88-7e140014dcaa",
    "answer": "...",
    "sources": [],
    "finish_reason": "stop"
  }
}
```

#### 12.3.2 会话列表

用途：
- 小程序查看当前用户的会话列表
- Go 可以直接查本地映射表，也可以必要时补查 MaxKB

### 12.4 推荐错误码映射
Go 建议统一翻译 MaxKB 错误：

- `400`：参数错误、缺少字段、格式不合法
- `401`：未登录、凭证失效、API Key 无效
- `403`：无权限访问该应用或知识库
- `404`：资源不存在
- `409`：资源冲突、重复创建
- `422`：业务校验失败，如应用未发布、未绑定模型
- `500`：上游异常、内部错误
- `502`：MaxKB 上游不可用或返回异常
- `504`：MaxKB 超时

- `application_id`

- `session_id`
- `message_id`
- `question`

- `sources`
- `finish_reason`
- `request_id` / `trace_id`
- `status`
- `error_message`
- `created_at`
- `updated_at`

### 12.6 推荐的前端调用方式
小程序调用 Go 时，建议统一走以下模式：

1. 小程序先拿到自己的登录态
2. 小程序调用 Go 的业务接口
3. Go 校验用户身份和租户权限
4. Go 选择可用的 `application_id`
5. Go 调用 MaxKB
6. Go 记录返回结果并返回给小程序

这样小程序永远不需要知道：
- MaxKB API Key
- MaxKB 内部接口地址
- MaxKB 的错误码细节

## 16. 最终结论

第三方对接建议采用以下模式：

- **MaxKB**：AI 能力服务
- **Go 后台**：业务网关、鉴权中台、审计中台
- **小程序**：交互展示层

从当前方案设计来看，**这套分层是合理的，且可以作为实施依据**。也就是说：
- MaxKB 侧负责知识库、文档、问答、会话能力
- Go 侧负责接入、权限、映射、审计、错误翻译
- 小程序侧负责 DeepSeek 风格的交互体验与展示

但要真正进入开发，仍建议在联调前补齐两类确认：
1. 真实 MaxKB 版本的接口路径、字段名、返回值语义
2. Go 侧对外 API 的最终契约与错误码映射

如果后续要做到更接近 DeepSeek 的效果，主要补的是**前端交互体验和流式渲染**，不是重构 MaxKB 核心。
