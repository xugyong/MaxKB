# MaxKB 第三方对接封板版

本文档定义 MaxKB 与第三方系统（Go 后台 + 小程序前端）的正式对接边界、稳定接口、验收标准与上线门槛。封板后，以下约定视为外部契约，不再随实现细节变化而随意调整。

---

## 1. 对接目标

第三方系统由两部分组成：

- **Go 管理后台**：统一鉴权、权限控制、审计日志、业务编排、调用 MaxKB
- **小程序前端**：仅负责展示与交互，不直接访问 MaxKB

MaxKB 作为内部 AI 能力服务，对外仅由 Go 后台调用。

---

## 2. 稳定对接原则

### 2.1 统一网关原则
所有第三方请求必须先到 Go 后台，再由 Go 后台调用 MaxKB。

### 2.2 凭证隔离原则
MaxKB API Key 只能保存在服务端，不得下发到小程序或浏览器。

### 2.3 身份分层原则
需要区分两类身份：

- **系统身份**：Go 后台调用 MaxKB 时使用的服务凭证
- **终端用户身份**：小程序用户身份，由 Go 后台映射到 MaxKB 会话

### 2.4 最小改动原则
MaxKB 保留为 AI/RAG 能力服务，业务权限、租户隔离、用户映射、审计日志尽量放在 Go 后台。

---

## 3. 稳定接口边界

### 3.1 MaxKB 对 Go 暴露的稳定接口
以下接口视为稳定对接接口：

- 健康检查
- 知识库创建 / 查询 / 更新 / 删除
- 文档上传 / 列表 / 详情 / 删除 / 重处理
- 应用 API Key 管理
- 问答接口
- 会话列表 / 消息列表 / 会话删除

### 3.2 Go 对小程序暴露的稳定业务接口
以下由 Go 自行定义并稳定对外：

- 创建知识库
- 上传文档
- 查询文档状态
- 发起问答
- 查询会话历史
- 查询业务可见的知识库列表

### 3.3 不作为外部契约的内容
以下内容不作为稳定契约，不建议第三方直接依赖：

- MaxKB 内部模型表结构
- 任务队列实现细节
- 文档解析器内部状态
- 会话复开内部逻辑

---

## 4. 认证与授权

### 4.1 Go 调用 MaxKB
Go 后台调用 MaxKB 时，使用：

```text
Authorization: Bearer <MAXKB_API_KEY>
```

### 4.2 小程序调用 Go
小程序只使用 Go 后台自己的登录态：

- JWT
- Session Cookie
- 小程序 code 换 session
- 其他你们已有的登录方式

### 4.3 权限分层
建议至少分三层：

- 管理员：创建知识库、上传文档、管理 API Key
- 普通用户：问答、查看自己的会话
- 业务租户：只能看到自己授权范围内的数据

---

## 5. 数据映射要求

Go 后台必须保存以下映射：

### 5.1 知识库映射
- 业务知识库 ID
- MaxKB `workspace_id`
- MaxKB `knowledge_id`
- 名称
- 状态
- 创建者

### 5.2 应用映射
- 业务应用 ID
- MaxKB `application_id`
- MaxKB API Key ID
- 启用状态

### 5.3 会话映射
- 业务用户 ID
- MaxKB `chat_id`
- MaxKB `message_id`
- 最近一次问答内容
- 最近一次回答内容

---

## 6. 初始化前提

封板前必须确保以下条件满足：

1. MaxKB 服务可正常启动
2. 至少存在一个可用的 `workspace`
3. 至少存在一个已发布的 `application`
4. 至少存在一个可用的 `application api key`
5. 至少存在一个可用的 embedding 模型
6. 知识库创建接口可用
7. 文档上传接口可用
8. 问答接口可用

---

## 7. 推荐调用流程

### 7.1 知识库创建
```text
管理员 -> Go 后台 -> MaxKB 创建知识库 -> Go 保存 knowledge_id
```

### 7.2 文档上传
```text
管理员 -> Go 后台 -> MaxKB 上传文档 -> Go 轮询状态 -> 前端展示结果
```

### 7.3 用户问答
```text
小程序 -> Go 后台 -> MaxKB chat/completions -> Go 返回答案与来源
```

### 7.4 会话历史
```text
小程序 -> Go 后台 -> 查询 Go 保存的映射 -> 必要时补查 MaxKB
```

---

## 8. 稳定返回约定

### 8.1 知识库创建
成功时必须返回：

- `code = 0`
- `data.workspace_id`
- `data.knowledge_id`
- `data.name`

### 8.2 文档上传
成功时必须返回：

- `code = 0`
- `data.workspace_id`
- `data.knowledge_id`
- `data.document_id`
- `data.status`

### 8.3 问答接口
成功时必须返回：

- `code = 0`
- `data.session_id`
- `data.message_id`
- `data.answer`
- `data.sources`
- `data.usage`

### 8.4 失败响应
失败时必须保留：

- `code`
- `message`
- `data`

并保证 Go 侧可将其映射为统一错误码。

---

## 9. 统一错误码建议

Go 后台建议统一映射如下：

- `401`：未登录或凭证无效
- `403`：无权限
- `404`：资源不存在
- `500`：MaxKB 内部处理失败
- `502`：MaxKB 上游不可用
- `504`：MaxKB 超时

## 11. Go 团队实施指南

这一节是给 Go 团队的最终交付说明。目标是让 Go 侧按统一契约完成接入、编排、治理与验收，并保证 MaxKB 核心保持稳定不被业务接入逻辑污染。

### 11.1 角色定位
Go 后台是第三方接入的唯一业务入口，负责：

- 统一鉴权与用户态管理
- 业务权限控制与租户隔离
- MaxKB 能力编排与稳定调用
- 统一错误映射与审计记录
- 对前端提供稳定、不暴露内部实现的接口

MaxKB 仅作为能力服务使用，不承载 Go 侧的业务会话治理、权限治理与审计主逻辑。

### 11.2 系统分层建议
Go 侧建议分为四层：

#### 11.2.1 Handler / Controller
职责：

- 接收请求
- 校验参数
- 校验登录态
- 转交 Service
- 统一返回 JSON

这一层不写 MaxKB 调用细节。

#### 11.2.2 Service
职责：

- 业务编排
- 调用 MaxKB Client
- 控制重试、超时、降级
- 错误码翻译
- 维护业务状态流转

#### 11.2.3 Client
职责：

- 封装 MaxKB HTTP 调用
- 处理 Base URL、Header、超时、重试
- 解析响应与错误
- 统一序列化与反序列化

#### 11.2.4 Repository / DAO
职责：

- 保存业务映射关系
- 保存会话与文档状态
- 保存审计日志与操作记录
- 保存告警与错误归因信息

### 11.3 必须实现的业务能力
Go 团队至少需要实现以下能力：

1. MaxKB Client 封装
2. 鉴权与凭证管理
3. 知识库创建与查询
4. 文档上传与状态查询
5. 问答与会话管理
6. 会话历史与消息历史
7. 审计日志与监控告警
8. 后台管理页面与权限控制

### 11.4 设计原则
#### 11.4.1 凭证只在服务端保存

- MaxKB API Key 只能保存在服务端
- 不允许下发给前端
- 支持轮换、停用、审计

#### 11.4.2 业务身份与 MaxKB 身份分离

- 业务用户 ID 不直接等同于 MaxKB ID
- 会话、文档、知识库需要建立映射表
- 前端只认 Go 侧业务 ID

#### 11.4.3 幂等优先

- 创建知识库、上传文档、发起问答建议支持幂等键
- 失败重试只针对网络与上游不可用场景
- 业务错误不盲目重试

#### 11.4.4 错误统一翻译

- 不把 MaxKB 原始错误直接透传给前端
- 统一转成业务错误码与可读文案
- 明确区分认证失败、权限不足、资源不存在、上游超时、上游异常

#### 11.4.5 可观测性优先

- 每次调用带 request_id / trace_id
- 所有异常落库或落日志
- 关键链路要有告警
- 关键状态必须可追踪

## 13. 最终结论

第三方对接的封板版建议采用以下模式：

- **MaxKB**：AI 能力服务
- **Go 后台**：业务网关、鉴权中台、审计中台
- **小程序**：交互展示层

这是当前最稳、最适合长期维护的对接方式。
